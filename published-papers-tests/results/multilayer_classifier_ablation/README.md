# Multi-layer classifier ablation (2026-05-05)

A systematic ablation of three retrieval improvements layered on top of the production pure-graph classifier (Spec 2). Run on the Friends benchmark (1,287 rows). 16 individual ablation runs across three phases plus a final-stack run.

**This folder is published in full — including the runs where the ideas didn't work.** Some findings are uncomfortable; they're documented honestly because that's the point of an ablation study.

---

## What this folder contains

- `RESULTS.md` — the final tables, what worked, what didn't, what we did about it. **Read this first.**
- `METHODOLOGY.md` — exact procedure, isolation contract, what was implemented, why we stopped where we did
- `phase_a_6_stage/` — six-stage matrix establishing the production-equivalent baseline + isolating each Layer 3 sub-improvement
- `sweeps_phases_b_c_d/` — adaptive-K threshold sweep (5 runs), hierarchical fallback sweep (4 runs), final stack (1 run)

Each subdirectory contains the raw `metrics.json` + `predictions.json` from each run, plus a top-level `summary.json` aggregating the phase. Anyone can re-score with the paper's `metrics.py` and reproduce the headline numbers.

---

## Headline result

The only sub-improvement that delivered a measurable gain over the production-equivalent baseline:

| Stage | Balanced accuracy | Δ vs production_equivalent |
|---|---:|---:|
| production_equivalent | 0.6464 | (anchor) |
| **prod + distance-weighted voting** | **0.6520** | **+0.56pp** |
| prod + adaptive K (best threshold) | 0.6360 | -1.04pp |
| prod + hierarchical retrieval | 0.5684 | -7.80pp |
| all three combined | 0.5628 | -8.36pp |

**Real, modest, defensible: +0.56pp from distance-weighted voting layered on top of production.** Adaptive K and hierarchical retrieval, as designed and tuned in this iteration, did not pay off on the Friends test set.

## What this folder is NOT

This is a **single benchmark on a single test set** of one architectural direction (Layer 3 of the multi-layer classifier roadmap). It is not:

- A claim that the multi-layer architecture works in production — the multi-layer code lives in `published-papers-tests/multilayer/`, isolated from `core/`, NOT integrated into the live KaraOS pipeline.
- A claim that distance-weighted voting is universally better — the result is specific to Friends + the production-equivalent baseline + the current scenario corpus mix.
- A complete validation of Layer 1 — Layer 1 (per-scenario learned reliability) requires Layer 2 (outcome supervision wiring) to fully test, and Layer 2 is deferred (it requires real-world data accumulation that doesn't fit our 5-week horizon).
- An apology for the negative results — Phase B and C show that adaptive K and hierarchical retrieval, in their current form, are past their useful operating range on this dataset. That's a finding, not a failure.

## Why this work happened

KaraOS's pure-graph classifier scored **0.6448 balanced accuracy** on the Friends benchmark from Bhagtani et al. 2026 — competitive with the lowest fine-tuned LoRA models in the paper (Qwen3-4B fine-tuned: 0.6512), above the human baseline (0.6375), and rare in the field for using zero LLM calls in the classification path.

A 10-run scaling ablation we published earlier ([`../friends_scaling_ablation/`](../friends_scaling_ablation/)) revealed an architectural limitation: **inverse scaling**. Adding more bootstrap scenarios to the retrieval database hurt accuracy on Friends, monotonically. The proposed diagnosis pointed at heterogeneous bootstrap-data quality: as the database grows, lower-quality scenarios crowd top-K and corrupt the vote.

The natural architectural response is a *multi-layer* classifier: per-scenario reliability scoring (Layer 1), outcome supervision (Layer 2), smarter retrieval (Layer 3), multi-aspect annotation (Layer 4), auto-quarantine (Layer 5), provenance tracking (Layer 6). The full roadmap is documented in `dog-ai/everything_about_system.md` Part XXXV.

This folder documents the implementation and ablation of **Phase 1 of that roadmap**: Layer 1's data infrastructure (the schema, the smoothed-prior multiplier) plus all three Layer 3 sub-improvements (hierarchical retrieval, adaptive K, distance-weighted voting). All built in a parallel codebase under `published-papers-tests/multilayer/` that doesn't touch production. We ran the ablation, measured what helped, and stopped — moving the +0.56pp finding into the production roadmap rather than rushing to integrate.

## Reproduction

Each subdirectory's `metrics.json` was generated by KaraOS's v2 classifier (`published-papers-tests/multilayer/classify_intent_graph_v2.py`) running against `published-papers-tests/dataset/friends/test/test_samples.jsonl`. To reproduce:

```bash
cd dog-ai/published-papers-tests/multilayer

# Phase A — 6-stage matrix
python -m scripts.run_ablation_v2 --phase a --out-path ../results/multilayer_ablation_phase_a/

# Phases B + C + D combined
python -m scripts.run_phases_b_c_d --out-path ../results/multilayer_ablation_sweep/
```

Total compute: ~16 minutes on a warmed-up GPU (E5-large-instruct as embedder, brute-force cosine k-NN over 2,071 scenarios). $0.00 actual cost — no API calls in the classification path.

## Honesty disclosure

Every number in this folder comes from a real run on real data. Where a sub-improvement didn't deliver — adaptive K (-1.04pp), hierarchical retrieval (-7.80pp), the all-three combined stack (-8.36pp), the parameter sweeps that produced identical numbers across all tested values — those results are published verbatim with their diagnoses attached.

If anyone reproducing the test gets a number that differs from what's reported here by more than ~1pp, please open an issue. We'll reconcile.
